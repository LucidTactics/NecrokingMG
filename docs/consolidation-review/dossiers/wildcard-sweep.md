## Finding 1 — Radial glow texture generated byte-for-byte in 3 places — CONSOLIDATE (severity: low, effort: S)

**Evidence (verified identical, 64x64 quadratic-falloff premultiplied radial glow):**
- `Necroking/Game1.cs:2308-2322` — `_glowTex` creation at startup
- `Necroking/Editor/EnvObjectEditorWindow.cs:833-850` — `EnsureGlowTex()` (comment: "same as Game1._glowTex")
- `Necroking/Editor/SpellPreview.cs:228-242` — comment: "matches Game1's _glowTex"

The loop bodies are byte-for-byte identical (same 31.5f center, quadratic falloff, premultiplied
`Color(a,a,a,a)`). Any tuning of the in-game glow (size, falloff exponent) silently desyncs the
two editor previews.

**Canonical home:** `Necroking/Render/TextureUtil.cs` — it already owns the analogous
device-keyed `GetWhitePixel(device)` cache (TextureUtil.cs:229-242).

**Merged API:**
```csharp
// TextureUtil — device-keyed cache, mirrors GetWhitePixel
public static Texture2D GetRadialGlow(GraphicsDevice device, int size = 64)
```

**Call sites to migrate:** Game1 startup (2308), EnvObjectEditorWindow.EnsureGlowTex (delete the
method, call the util), SpellPreview init (229). Risk: trivial — pixel output identical; caching
across the three users is strictly a win (they never dispose it independently today — verify
SpellPreview doesn't Dispose its copy; if it does, drop that Dispose).

## Finding 2 — 2.5D world→screen projection formula re-implemented outside Camera25D — CONSOLIDATE (severity: medium, effort: S)

**Evidence:**
- Canonical: `Necroking/Render/Camera25D.cs:33-48` — `WorldToScreen` / `WorldToScreenPx`
- Copy: `Necroking/Render/Renderer.cs:45-59` — `WorldToScreen` / `WorldToScreenPx` re-implement the
  identical formula bodies (only difference: `_screenW/_screenH` fields vs parameters) instead of
  delegating to the camera
- Inline copy: `Necroking/Render/GrassTuftRenderer.cs:319-320` — `DrawSingleTuft` computes
  `sx/sy` by hand (no-height special case of the same formula)
- Related: the view-frustum bounds calc (`viewLeft/Right/Top/Bottom` from camera pos, zoom,
  YRatio, pad) is duplicated in `DeathFogRenderer.AddPuffsToDepthList` (DeathFogRenderer.cs:89-92)
  and `GrassTuftRenderer` (GrassTuftRenderer.cs:215-218).

This is the game's core 2.5D projection (YRatio foreshortening). It is currently in sync, but a
change to the projection (e.g. YRatio semantics, camera shake offset) requires 5+ coordinated
edits, and the inline GrassTuft copy would be the one that gets missed.

**Canonical home:** `Camera25D` (it owns Position/Zoom/YRatio).

**Merged API (no new types):**
```csharp
// Renderer.cs — delegate instead of re-implementing:
public Vector2 WorldToScreen(Vec2 p, float h, Camera25D cam) => cam.WorldToScreen(p, h, _screenW, _screenH);
public Vector2 WorldToScreenPx(Vec2 p, float px, Camera25D cam) => cam.WorldToScreenPx(p, px, _screenW, _screenH);
// Camera25D — new: view bounds used by culling renderers
public (float l, float t, float r, float b) ViewBounds(int screenW, int screenH, float pad);
```

**Call sites:** Renderer's two methods (bodies only — external callers unchanged);
GrassTuftRenderer.DrawSingleTuft; DeathFogRenderer + GrassTuftRenderer culling bounds.
Risk: low — pure formula substitution, verify with a screenshot pass (`/drive-game`).
Renderer/ScreenToWorld pair (Renderer.cs:61-65 vs Camera25D.cs:50-54) same treatment.

## Finding 3 — Y-sorted "puff layer" pattern mirrored across three renderers (+ duplicated hash helpers) — INVESTIGATE (severity: low)

**Evidence:**
- `Necroking/Render/DeathFogRenderer.cs:43-51,66-166` — `PuffData` struct + `AddPuffsToDepthList`
  + `DrawSinglePuff` (doc comment at 40-42 explicitly says "Mirrors PoisonCloudRenderer.PuffData")
- `Necroking/Render/PoisonCloudRenderer.cs:20-31,80-160` — identical `PuffData` fields, identical
  `DrawSinglePuff` body (two-level [cloud][puff] cache instead of flat list)
- `Necroking/Render/GrassTuftRenderer.cs:255-329` — same collect-then-draw-by-index pattern with
  a different instance struct (`TuftInstance`: texture, flip, deferred projection)
- Exact helper duplicates: `HashToFloat` byte-identical at DeathFogRenderer.cs:176 and
  GrassTuftRenderer.cs:351; `CellHash` (DeathFogRenderer.cs:168, 2-arg) vs `TileHash`
  (GrassTuftRenderer.cs:343, 3-arg) are the same mix with one extra term.

**Why not a clean CONSOLIDATE:** the *mechanics* that repeat are small (a cached list, a bounds
check, a 5-line `spriteBatch.Draw`), while the *generation* logic — where all the code mass is —
is genuinely structural variance: density-grid iteration vs ring-noise-around-entity vs per-tile
tuft scatter, and each has its own deliberate sortY convention (southern-extent vs feet-anchor,
documented per file). Per CLAUDE.md, abstracting that would create a framework, not a utility.
The mirroring is *acknowledged* in doc comments, which halves the divergence risk.

**Decision needed:** (a) extract a minimal `DepthSpriteLayer` (owns `List<PuffData>`,
`DrawSingle(SpriteScope, int)`, `AddDepthItem(y, type, idx)`) and keep generation per-renderer;
or (b) accept the documented mirroring. Either way, the hash helpers should move to one small
`Render/HashNoise.cs` (or Core) — that part is a free S-effort win and keeps DeathFog's jitter
stable (same constants).

## Finding 4 — WadingWakeSystem entry/exit splash session accounting duplicated — CONSOLIDATE (severity: low, effort: S)

**Evidence (all in `Necroking/Render/WadingWakeSystem.cs`, single file, private statics):**
- `StartEntrySplash` (1261-1285) vs `StartExitSplash` (1460-1488): identical structure — compute
  total from speed, split burst/trickle, cache session state, emit burst — on parallel state
  field sets (`Splash*` vs `Exit*` on `WakeEmitterState`).
- `TrickleEntrySplash` (1292-1328) vs `TrickleExitSplash` (1495-1523): the accumulator loop is
  copy-paste line-for-line (rate = remaining/duration; accum += rate*dt; spawn; decrement),
  differing only in which state fields it touches. A fix to the accumulator rounding/catch-up
  logic in one would need mirroring in the other.
- `EmitSplashParticles` (1336-1452) vs `EmitExitSplashParticles` (1530-1624): **genuinely
  different physics** (upward fountain + forward cone vs downward drips + height fraction +
  LandHeight=0). These should NOT be merged.

**Canonical design (mechanics/data split):** a `SplashSession` struct owning the accounting:
```csharp
struct SplashSession { public float SpeedAtStart, RemainingDuration, SpawnAccum; public int RemainingCount;
    public void Start(int total, float burstFrac, float duration, float speed, out int burst);
    public int Tick(float dt); } // returns toSpawn
```
`WakeEmitterState` holds two `SplashSession` fields (entry, exit); Start/Trickle pairs collapse to
thin wrappers that call the session then their own Emit*. Effort S, risk low (visual-only —
verify with a wade-in/wade-out `/drive-game` check).

## Finding 5 — Premultiply-alpha loop duplicated 3x inside TextureUtil — CONSOLIDATE (severity: low, effort: S)

but the Timed copy is pure debt.

## Finding 6 — RuntimeWidgetRenderer Set*Override family: 7 get-or-create-dict setters already drifting — CONSOLIDATE (severity: low, effort: S)

## Finding 8 — Editor preview shadow re-derives the in-game shadow model — INVESTIGATE (severity: low)

**Evidence:** `Necroking/Editor/EnvObjectEditorWindow.cs:759-831` (`DrawPreviewShadow`) hardcodes
`sOpacity=0.35f, sLengthScale=0.6f, sSquash=0.3f, sSunAngle=225f` with the comment "Defaults
match data/settings.json ShadowSettings", and approximates the in-game BasicEffect parallelogram
skew with a rotation+squash draw. The doc comment (755-757) *explicitly acknowledges* the
divergence: "if the user tunes the global shadow settings the preview won't reflect it".

**Why not CONSOLIDATE outright:** the code documents a real constraint — the in-game
ShadowRenderer path requires ending the SpriteBatch (BasicEffect quad), which would drop the
editor's scissor state. The approximation is a deliberate trade-off, not accidental copy-paste.

**Decision needed:** whether the editor should (a) read live `ShadowSettings` values instead of
copied constants (cheap, kills the constant-drift half of the problem), and (b) whether the lean
angle formula (815-816) should be exposed from the shadow system so preview and game share the
math even if the rasterization differs. (a) looks like a clear small win; (b) needs a look at
ShadowRenderer's exact formula first.

## Finding 9 — AnimRequest factory family — KEEP_SEPARATE

`Necroking/Render/AnimController.cs:159-196`: `Locomotion/Action/Reaction/Combat/Forced/Hold`
are six named policy factories on one struct. Each encodes a distinct, documented
priority/interrupt/lifecycle combination (the doc block at 135-147 defines the priority layering;
`Reaction`'s comment even routes callers through DamageSystem gates). Collapsing them into one
parameterized constructor would delete the semantic layer that prevents callers from
hand-assembling wrong priority/Kind combos. This *is* the consolidation — the labeler's
suggestion is backwards.

## Finding 10 — SpriteScope Draw/DrawString overload family — KEEP_SEPARATE

## Finding 11 — Misc labeler over-matches — KEEP_SEPARATE

- **`Locomotion.IsLocoClass` vs `AnimController.IsLocomotionState`** (Locomotion.cs:320-322,
  AnimController.cs:529-530): labeler claimed "duplicates exactly" — **false**. IsLocoClass
  includes `Idle` (gait selection treats standing as a loco class); IsLocomotionState excludes it
  (phase-preserving transitions only apply between moving gaits). Different predicates with
  different semantics; merging would introduce a bug.
- **SkillBookState unlock stores** (SkillBookState.cs:102-110): `UnlockPotion`/`SetPassive`/
  `UnlockSummon` are one-line adds to *typed domain sets* with per-domain guards (UnlockSummon
  null-checks, UnlockAI is monotone). A generic flag store would erase the domain API for zero
  code savings. `LearnFree` vs `TryLearn` (278-320) differ structurally (cost deduction +
  rollback control flow) — exactly the "structural variance" CLAUDE.md says not to abstract.
- **Modal Close patterns** (WallEditorWindow.cs:115-119, ColorPickerPopup.cs:191+,
  EnvObjectEditorWindow.cs:253-259, etc.): the shared mechanics already live in the canonical
  `PopupManager`/`IModalLayer` (UI/PopupManager.cs:23); each Close resets its own fields (caller
  owns data) and pops. This is the pattern working as designed. One inconsistency worth a
  note: `WadingEditorPopup` never registers with `Game1.Popups` at all (no Push/Pop in the file)
  — if it's meant to be modal, that's a gap, but it's a behavior question, not duplication.
- **`TriggerRegion.ContainsPoint` vs `MapZone.ContainsPoint`** (TriggerTypes.cs:21-26,
  ZoneTypes.cs:61-62): one-line rect containment on two independent domain structs (trigger also
  supports circles). One shared line of math is below the abstraction threshold.
- **WeatherRenderer.Init/Resize** (WeatherRenderer.cs:32-42): two identical 2-line methods that
  are semantic aliases; harmless. Could make Init call Resize; not worth tracking.
- **Editor keyboard dispatchers** (EnvObjectEditorWindow.HandleKeyboard vs ItemEditor
  HandleKeyboardShortcuts): per-window hotkey switch statements — structural variance (each
  dispatches to its own actions); a shared dispatcher would be a framework.
